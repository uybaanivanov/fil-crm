# Источники броней, легенда шахматки, кликабельные сводки, кнопка «Сделать договор»

Дата: 2026-04-23
Статус: design (approved)

## Цель

Четыре независимых, но небольших улучшения CRM:

1. **Источники броней** — перенести `source` с клиента на бронь, расширить список каналов
   (Авито, Островок, Суточно.ру, Доска.якт, Юла, Фил).
2. **Шахматка** — в левой колонке вместо позывного показывать сокращённый адрес.
3. **Сводки/недавние** — везде, где бронь отображается в виде строки/карточки, клик ведёт
   на карточку этой брони.
4. **«Сделать договор»** — кнопка в карточке брони (для owner+admin), модалка с выбором
   типа гостя, WIP-экран. Без бэка, ничего не сохраняем.

## Контекст

- БД: SQLite, raw-запросы (без ORM). `db.sqlite3` локально, `/opt/fil-crm/db.sqlite3` на проде.
- Backend: FastAPI, миграции в `backend/migrations/NNN_*.sql`.
- Frontend: SvelteKit (Svelte 5).
- Сейчас `clients.source` хранит «как клиент попал в базу впервые», в форме новой брони
  селект источника спрятан внутри блока «создать клиента». Список значений: `Доска.якт`,
  `Юла`, `Прямая`. Семантика «источник на клиенте» неверна для повторных гостей через
  разные каналы, плюс нужно расширить список.

## 1. Источники броней (`bookings.source`)

### Миграция `010_booking_source.sql`

```sql
ALTER TABLE bookings ADD COLUMN source TEXT;

UPDATE bookings
SET source = (SELECT source FROM clients WHERE clients.id = bookings.client_id)
WHERE source IS NULL;

ALTER TABLE clients DROP COLUMN source;
```

SQLite ≥ 3.35 поддерживает `DROP COLUMN`. В проекте версия современная (см. `pyproject.toml`,
системный sqlite WSL Ubuntu — 3.40+). Если на проде SQLite окажется старее — фоллбэк на
table-rebuild (`CREATE TABLE clients_new AS SELECT … FROM clients; DROP; RENAME`). Решение —
в момент имплементации после проверки `SELECT sqlite_version()` на проде.

### Допустимые значения

Хранится текстом, **без CHECK** в БД (свобода добавлять каналы без миграций).

Канонический список (строго в этом порядке) — `frontend/src/lib/sources.js`:

```js
export const BOOKING_SOURCES = [
    'Авито',
    'Островок',
    'Суточно.ру',
    'Доска.якт',
    'Юла',
    'Фил',
];
```

«Фил» = ручной канал, заказчик добавил бронь сам (то, что раньше называлось «Прямая»).

Бэк дополнительной валидации списка не делает.

### Backend (`backend/routes/bookings.py`)

- `BookingIn`: добавить `source: str | None = None`.
- `BookingPatch`: добавить `source: str | None = None`.
- `_row()` SELECT: добавить `b.source`.
- `list_bookings()` SELECT: добавить `b.source`.
- `bookings_calendar()` — `source` не нужен на шахматке, не добавляем.
- `create_booking`: пишет `source` (новый параметр в INSERT).
- `update_booking`: уже работает обобщённо через `model_dump(exclude_unset=True)` —
  добавление поля в Pydantic-модель достаточно.

### Backend (`backend/routes/clients.py`)

- Убрать `source` из `ClientIn`, `ClientPatch`, всех SELECT и INSERT.

### Frontend

**`/bookings/new/+page.svelte`:**

- Новая секция «Источник» (между «Гость» и «Сумма»). Один `<select>` со значениями из
  `BOOKING_SOURCES`. Обязательный — валидация на сабмите (`if (!source) return error('…')`).
- Из inline-формы «Создать нового клиента» **убрать** поле «Источник» и `newSource`.
- При POST `/bookings` передавать `source`.

**`/bookings/[id]/+page.svelte`:**

- В шапке `PageHead.sub` заменить `client?.source || 'Без источника'` на
  `booking?.source || 'Без источника'`.
- В chip-row рядом со статусом — добавить отдельный chip с источником
  (`<Chip tone="info">{booking.source}</Chip>`), если он задан.

**`/clients/[id]/+page.svelte`:**

- Убрать строку «источник: …» (`data.source` больше нет).

**Прочие места, где использовался `client.source`:**

- `/clients/+page.svelte` — проверить, нет ли группировки/фильтрации по `source`. Если есть — снести.
- Любые другие упоминания — найти `grep -rn "\.source" frontend/src/` и почистить.

### Совместимость и риски

- Старые брони без `source` после data-migration получат значение из `clients.source` (если
  оно было). У свежесозданных клиентов источник не назначался — у их броней `source = NULL`,
  в UI отобразится «Без источника». Можно вручную проставить через `PATCH /bookings/{id}`.
- Сложность: **норм** (миграция, контракт API, форма). Риск регрессий — низкий, source
  никем не валидируется логически (просто отображается).

## 2. Шахматка: легенда — сокращённый адрес

### Backend (`backend/routes/bookings.py::bookings_calendar`)

- В SELECT по апартаментам добавить `a.address`.
- В bucket добавить ключ `apartment_address`.
- `apartment_callsign` оставить (не удалять — может пригодиться для других экранов в будущем).

### Frontend (`frontend/src/routes/calendar/+page.svelte`)

- Импортировать `shortAddress` из `$lib/format.js`.
- Параллельный запрос `api.get('/apartments')` **убрать** — список квартир строим из
  `calendar` (бэк уже отдаёт по строке на квартиру). Экономим один запрос.
- В левой колонке (`name-col`) показывать `shortAddress(apartment_address) || apartment_title`.
- Атрибут `title=` на `name-col` — полный адрес (для tooltip).
- Класс `.cs` (моно-стиль для callsign) больше не используется — удалить из CSS.
- Шапка колонки («Апрель» — заголовок таблицы) не меняется.

### `frontend/src/lib/format.js` — новая функция

```js
export function shortAddress(addr) {
    if (!addr) return '';
    let s = addr;
    s = s.replace(/^(ул\.?|улица|пр\.?|проспект|пер\.?|переулок|пл\.?|площадь|ш\.?|шоссе|б-р|бульвар)\s+/i, '');
    s = s.replace(/,?\s*(кв\.?|квартира)\s*\d+\S*/i, '');
    s = s.replace(/,?\s*(под\.?|подъезд)\s*\d+/i, '');
    s = s.replace(/\s+/g, ' ').trim();
    return s;
}
```

Примеры:

| Полный адрес                            | Сокращённый           |
|-----------------------------------------|-----------------------|
| ул. Петра Алексеева, 50, кв. 34         | Петра Алексеева, 50   |
| пр. Ленина 10/2 кв 18                   | Ленина 10/2           |
| Ойунского 7                             | Ойунского 7           |

### Сложность: **легко**

Один файл фронта + один SELECT на бэке + новая утилита.

## 3. Кликабельные брони в сводках

### 3.1 `/apartments/[id]/+page.svelte` — блок «Гость»

Сейчас одна кнопка `.guest` ведёт на клиента, ниже статичный `.guest-range` с датами/суммой.

После изменения:

- `.guest` (аватар + имя + телефон) → клик ведёт на `/clients/{client_id}` (как сейчас).
- `.guest-range` (даты + сумма) → становится `<button>`, клик ведёт на
  `/bookings/{booking_id}`.
- Визуально между ними разделитель `border-top: 1px solid var(--border-soft)`.
- Hover для обеих зон: `background: var(--card-hi)`.
- Cursor: pointer для обеих.

### 3.2 `/finance/+page.svelte` — секция «Последние движения»

Бэк (`backend/routes/finance.py`) уже отдаёт `feed[].ref` — для income строк там
`{booking_id}`, для expense строк `{expense_id}`.

После изменения:

- `feed-row` рендерим как `<button>`, если есть `ref.booking_id` или `apartment_id` для
  expense. Клик:
  - `ref.booking_id` → `goto(/bookings/{booking_id})`.
  - `ref.expense_id` + `apartment_id` → `goto(/apartments/{apartment_id})` (отдельной
    карточки расхода нет; идём на квартиру).
  - Расход без `apartment_id` (общий расход) → не кликабелен, остаётся `<div>`.
- Стили: добавить `:hover { background: var(--card-hi); }` и `cursor: pointer` для
  кликабельных строк.
- Если `<button>` ломает grid-layout строки — фоллбэк на `<div role="button" tabindex="0">`
  с `onkeydown` (Enter → переход). Предпочтение `<button>`.

### Сложность: **легко**

Только фронт, два файла.

## 4. Кнопка «Сделать договор» + модалка + WIP

### Размещение

В `/bookings/[id]/+page.svelte`, новая секция **«Договор»** между «Заметкой» и `Actions`.

Видна только если `me?.role === 'owner' || me?.role === 'admin'` (стандартный паттерн в
проекте).

```svelte
{#if me?.role === 'owner' || me?.role === 'admin'}
    <Section title="Договор">
        <div class="wrap">
            <button class="ghost" type="button" onclick={() => contractModalOpen = true}>
                Сделать договор
            </button>
        </div>
    </Section>
{/if}
```

Стиль кнопки — `.ghost` (как «Отменить» в actions), чтобы не конкурировать с primary
«Закрыть бронь».

### Будущая семантика (закладываем, не пишем сейчас)

Когда появится бэк генерации:

- POST `/bookings/{id}/contract` с `{kind: 'business' | 'personal'}` стартует генерацию.
- Фронт поллит статус.
- Если у брони есть готовый документ (URL/blob) — секция показывает ссылку «Скачать
  договор», а не кнопку.
- Если документа нет — кнопка «Сделать договор» (текущее состояние всегда).

В коде сейчас комментариев на эту тему **не пишем** (правило проекта). Структура секции
просто оставляет место для будущей замены контента.

### Модалка `frontend/src/lib/ui/ContractModal.svelte` (новый компонент)

Базовый паттерн модалки берём из `CleaningDueDialog.svelte` (если в проекте есть
переиспользуемый `Modal`/`Dialog` — используем его).

**Состояния (`$state`):**

```js
let step = $state('choose'); // 'choose' | 'wip'
let chosenKind = $state(null); // 'business' | 'personal' — для логов в будущем, сейчас не используется
```

При `open` → false сбрасываем `step = 'choose'`. Эффект подписывается только на `open`,
пишет только в `step` — соблюдаем правило проекта (не читать в эффекте тот state, в
который пишем).

**Шаг `choose`:**

- Заголовок: **«Для кого делаем договор?»**
- Кнопка «×» сверху-справа — закрывает модалку.
- Две большие кнопки:
  - **Командированный** → `chosenKind = 'business'; step = 'wip'`
  - **Не командированный** → `chosenKind = 'personal'; step = 'wip'`
- На мобиле — в столбик, на десктопе — в ряд (через `flex-direction: column` + media query
  при необходимости; в проекте мобила — основной форм-фактор, начинаем с column).

**Шаг `wip`:**

- Заголовок: **«Скоро»**
- Текст:
  > Генерация договора скоро появится. Сейчас эта кнопка — заглушка: выбор сохранён,
  > сам документ подтянем позже.
- Кнопка «Понятно» внизу — закрывает модалку.

Примечание: фраза «выбор сохранён» — формулировка для заказчика; по факту ничего не
сохраняем. Сознательный «лояльный» текст. Если автору это неприемлемо — заменить на
«Когда генерация будет готова, ты сможешь выбрать тип ещё раз и получить документ.»
(пока оставляем первый вариант — был согласован).

### Сложность: **легко**

Один новый компонент + одна секция в карточке брони. Никакой логики, никакого state на
сервере.

## Что НЕ делаем (вне скоупа)

- Бэкенд для генерации договоров (POST `/bookings/{id}/contract`, polling, файлы) — отдельная
  работа, появится позже.
- Колонка `bookings.contract_type` в БД — не создаём, выбор не сохраняем.
- Переезд `clients.source` в отдельный `client_acquisition_source` — нет, удаляем
  безвозвратно (после переноса значений в `bookings.source`).
- Любые изменения в `/clients/+page.svelte`, кроме удаления упоминаний `client.source`,
  если они есть.

## Тестирование

- Backend: pytest для `bookings.py` (создание/патч с source, миграция данных при rollout
  проверяется вручную на копии БД).
- Frontend: проверить вручную:
  - Создание брони с источником.
  - Карточка брони показывает источник в шапке и chip.
  - Шахматка показывает сокращённые адреса; tooltip — полный.
  - Клик на даты в блоке «Гость» на карточке квартиры → бронь.
  - Клик на income/expense в /finance → бронь / квартира.
  - Кнопка «Сделать договор» только у owner/admin; модалка открывается, обе кнопки ведут
    на WIP, кнопка «Понятно» закрывает, повторное открытие начинается с шага `choose`.

## План rollout

1. Миграция `010_booking_source.sql` — локально, проверка данных.
2. Backend изменения (bookings.py, clients.py).
3. Frontend изменения (новый список источников, форма брони, карточка брони, шахматка,
   кликабельные строки).
4. ContractModal + секция «Договор» в карточке брони.
5. Прогон вручную локально.
6. Перенос на прод (`/opt/fil-crm`): миграция → перезапуск бэка → пересборка фронта.
