# Inline-редактирование полей карточки квартиры

## Проблема

На карточке квартиры (`/apartments/[id]`) все поля паспорта (название, адрес, тип, площадь, этаж, район, цена/ночь, позывной, обложка) сейчас только read-only в секции «Характеристики». Чтобы что-то поменять, приходится лезть в бэкенд вручную. Нужна возможность редактировать эти поля прямо из карточки — быстро, без перехода на отдельный экран.

Визуальный сигнал «это поле можно редактировать» — серый пунктир снизу, аналогично тому что уже сделано для обязательных `monthly_rent` / `monthly_utilities` (там цвет `primary`, а здесь — обычный `var(--border)`).

## Решения

### UX-паттерн: inline per-field

- В «показ»-режиме значение отображается как текст с `border-bottom: 1px dashed var(--border)`. Ховер усиливает цвет. Видно, что поле редактируемое.
- Клик по значению → оно in-place заменяется на `<input>` того же размера, с фокусом и выделенным текстом.
- **Enter** или **blur при изменённом значении** → `PATCH /apartments/{id}` с одним полем.
- **Escape** → откат к исходному значению, без запроса.
- **Blur без изменений** → просто возврат в «показ»-режим без запроса.
- Во время запроса — лёгкое затемнение/disabled. По успеху значение обновляется локально, refetch не делается.
- При ошибке 400/409 — красный `border-bottom` + сообщение-тултип снизу (с текстом из `detail`), автоматически возвращается в режим редактирования чтобы юзер мог поправить.
- Null-значение (например, `district` пуст) — в «показ»-режиме показан `placeholder` курсивом в цвете `var(--faint)`.

### Новый компонент `InlineEdit.svelte`

Путь: `frontend/src/lib/ui/InlineEdit.svelte`

Пропсы:
- `value: string | number | null` — текущее значение.
- `type: 'text' | 'number'` — тип инпута.
- `placeholder: string` — показ в «показ»-режиме, если `value == null`.
- `format?: (v) => string` — опциональная функция форматирования для отображения (например, `fmtRub` для цены, `v => v + ' м²'` для площади).
- `onSave: (newValue) => Promise<void>` — коллбек сохранения; если throw/reject — инлайн-ошибка.
- `readonly?: boolean` — если `true`, показывает только текст без пунктира и без возможности клика.

Внутри:
- Локальный стейт: `editing: boolean`, `draft: string | number`, `saving: boolean`, `error: string | null`.
- При входе в режим редактирования — копирует `value` в `draft`, ставит фокус.
- `onEnter / onBlur(modified) → save()`:
  - `saving = true`
  - `await onSave(draft)`
  - по успеху: `editing = false`, `saving = false`, `error = null`
  - по ошибке: `saving = false`, `error = <сообщение>`, остаётся в режиме редактирования.
- `onEscape` или клик вне (с учётом отсутствия изменений) — откат.
- Для числового типа — `parseFloat`/`parseInt` перед отправкой; если NaN → локальная ошибка «введи число».

### Интеграция в карточку квартиры

Файл: `frontend/src/routes/apartments/[id]/+page.svelte`

1. В `PageHead`-секции: `title` и `address` (подзаголовок) заменить на `InlineEdit`. `PageHead` принимает `title` и `sub` как строки — нужна минимальная доработка: либо поддержка слотов в `PageHead`, либо оборачивание руками рядом. Проще — оставить `PageHead` как декоративный заголовок, а `InlineEdit` для `title`/`address` положить отдельно под ним, стилизовав под крупный текст. (Решение: оставляем `PageHead` как «статичное отображение имени», а редактируемые заголовки добавляем внутри первой секции карточки, чтобы не переписывать `PageHead`.)

   **Уточнение:** чтобы не раздувать задачу — `title` и `address` редактируем **через ту же строку в «Характеристиках»**, а `PageHead` продолжает показывать снапшот (обновится при следующем `load()` или можно сделать `apt.title` реактивным; достаточно локально обновить `apt` после успеха `onSave`).

2. В секции «Характеристики» каждую строку переписать с использованием `InlineEdit`. Список полей:

   - **Название** (`title`): text, без формата.
   - **Адрес** (`address`): text.
   - **Тип** (`rooms`): text, placeholder «Напр. 1-комн».
   - **Площадь** (`area_m2`): number, format `v => v + ' м²'`.
   - **Этаж** (`floor`): text.
   - **Район** (`district`): text, placeholder «Район».
   - **Цена/ночь** (`price_per_night`): number, format `fmtRub`.
   - **Позывной** (`callsign`): text, placeholder «позывной».
   - **Обложка** (`cover_url`): text, placeholder «URL картинки». При успехе сразу обновляется превью (оно привязано к `apt.cover_url`).

3. **«Нужна уборка»** — не заменяем на `InlineEdit`. Редактируется через существующий `CleaningDueDialog` как раньше.

4. Для каждого поля один общий `onSave`-хелпер:

```javascript
async function patchField(field, value) {
    await api.patch(`/apartments/${aptId}`, { [field]: value });
    apt[field] = value; // локальное обновление без refetch
}
```

Использование: `<InlineEdit value={apt.price_per_night} type="number" format={fmtRub} onSave={v => patchField('price_per_night', v)} />`

### Пермишены

- `owner` и `admin` — компонент в редактируемом режиме (с пунктиром).
- `maid` — в `[id]/+page.svelte` пропускаем проп `readonly=true` в `InlineEdit`. В этом режиме пунктира нет, клик не открывает инпут. Текущий статус: карточка квартиры уже закрыта для `maid` (`GET /apartments/{id}` требует owner/admin), но если это изменится в будущем — пропс `readonly` готов.

### Обработка ошибки «baseline обязателен»

Сервер (`PATCH /apartments/{id}`) отклоняет любые патчи, если `monthly_rent` или `monthly_utilities` в итоговом состоянии остаются NULL, с `400 detail="monthly_rent и monthly_utilities обязательны"`.

Это делает редактирование, скажем, названия невозможным пока baseline не заполнен. В `InlineEdit` будет отображаться сообщение от сервера, но пользователю будет непонятно почему. Поэтому:

- При получении именно этой ошибки (детект по подстроке `monthly_rent` в сообщении) — показываем баннер сверху карточки с текстом «Сначала заполни обязательные суммы» и автоматически скроллим к секции «Обязательные суммы».
- Баннер исчезает после успешного сохранения baseline.

Реализация: `[id]/+page.svelte` хранит `let baselineRequired = $state(false)`. `patchField` ловит ошибку и, если в `error.detail` есть `monthly_rent`, выставляет `baselineRequired = true` и скроллит. После `saveBaseline()` (уже существует) — `baselineRequired = false`.

## Что НЕ делаем (YAGNI)

- Не переписываем `PageHead` — заголовок там остаётся статическим отображением (`title` и `address`). Их редактирование — в секции «Характеристики», после успеха локальное `apt.title = ...` обновляет UI везде где он используется (в `PageHead` тоже, т.к. Svelte reactivity).
- Не делаем «Нужна уборка» через `InlineEdit` — у неё своя datetime-логика и диалог, он работает.
- Не делаем валидацию на клиенте сверх базовой («число/не число»). Полагаемся на серверные `Field(gt=0)` / `min_length=1`.
- Не делаем undo/history.
- Фронтовых юнит-тестов не пишем (их в проекте нет).

## Тестирование

**Бэк:** изменений нет, существующие тесты `test_apartments.py` покрывают PATCH + валидацию baseline.

**Фронт:** ручной смок после деплоя:
1. Открыть квартиру с заполненным baseline. Кликнуть по «Цена/ночь», ввести новое, Enter. Значение обновилось, refetch не потребовался.
2. Кликнуть, изменить, Escape — значение откатилось.
3. Кликнуть, ввести пустую строку в обязательное поле (title) и Enter — красная рамка + сообщение от сервера, инпут остался в режиме edit.
4. Открыть квартиру БЕЗ baseline. Попытаться изменить название — появляется баннер «Сначала заполни обязательные суммы», скролл до блока baseline. Заполнить baseline → баннер уходит → повторно можно редактировать название.
5. Изменить `cover_url` на новую картинку — превью в шапке обновилось.

## Out of scope

- Клиентская валидация форматов URL/координат.
- Тултипы/подсказки per-field.
- Массовое редактирование (выделить N полей и менять одной кнопкой).
- Мобильная UX с модалкой вместо inline (мобилка работает с тем же `InlineEdit` — десктоп-клик = тап).
